## 一、严重 Bug (Critical)

### 1.1 缺少 CSRF 保护
**严重度**: Critical
**描述**: 所有表单（登录、注册、支付）均无 CSRF Token 保护，攻击者可构造恶意页面诱导用户提交表单。
**影响**: 用户可能在不知情的情况下被提交订单、修改信息等。
**复现**: 查看任意表单 HTML，均无 _csrf / token / nonce 隐藏字段。

### 1.2 安全响应头全部缺失
**严重度**: Critical
**描述**: 所有页面均缺少以下安全响应头：
- X-Frame-Options — 可被 iframe 嵌入，存在 Clickjacking 风险
- Content-Security-Policy — 无内容安全策略，XSS 攻击面大
- X-Content-Type-Options — 缺少 MIME 类型嗅探保护
- X-XSS-Protection — 缺少浏览器 XSS 过滤器
- Strict-Transport-Security — 缺少 HSTS
**复现**: 请求任意页面检查响应头。

### 1.3 支付接口无输入验证
**严重度**: Critical
**端点**: POST /cart/pay
**描述**: 支付接口接受以下恶意输入且均返回 HTTP 200：
- 空字段（username='', phone='', address=''）
- 负数金额（total=-100）
- SQL 注入（username="';DROP TABLE--"）
- XSS 攻击（username='<script>alert(1)</script>'）
- 修改隐藏字段 total 可任意篡改支付金额

**复现步骤**:
1. 登录后添加商品到购物车
2. 修改结算弹窗中 <input type="hidden" name="total"> 的值
3. 提交后订单以篡改金额创建成功

### 1.4 购物车数量无验证
**严重度**: Critical
**端点**: GET /cart/addToCart?goodId=X&quantity=Y
**描述**: 数量参数接受非法值且不报错：
- quantity=0 -> 200 (可添加0件商品)
- quantity=-1 -> 200 (可添加负数)
- quantity=999999 -> 200 (超大数量)
- quantity=abc -> 200 (非数字)
- quantity=  (空) -> 200

### 1.5 下单页面 404
**严重度**: Critical
**端点**: GET /order/orderAdd
**描述**: 虽然结算通过购物车弹窗 POST 到 /cart/pay 完成，但直接访问 /order/orderAdd 返回 404。
**影响**: URL 不一致，且如果弹窗 JS 加载失败，用户无法完成下单。

## 二、高危 Bug (High)

### 2.1 不存在的商品返回 200 而非 404
**严重度**: High
**端点**: GET /goods/detail?id=999
**描述**: 不存在的商品 ID（0、-1、999）返回 HTTP 200 并显示"商品不存在"页面，应返回 HTTP 404。
**影响**: SEO 问题，搜索引擎会索引大量"商品不存在"页面。

### 2.2 已登录用户仍可访问登录页
**严重度**: High
**端点**: GET /user/login
**描述**: 已登录用户访问 /user/login 仍显示登录表单，应自动跳转到首页。
**复现**: 登录后访问 /user/login，标题显示"登录 — CakeShop"。

### 2.3 管理员页面权限控制不一致
**严重度**: High
**端点**: /admin/*
**描述**:
- 未登录访问 -> 跳转登录页（HTTP 200）
- 普通用户登录后访问 -> HTTP 404
**问题**: 普通用户应收到 HTTP 403 Forbidden 而非 404 Not Found，否则用户无法区分"页面不存在"和"无权限"。

### 2.4 退出登录 URL 不一致
**严重度**: High
**描述**: 导航栏使用 /user/loginout，但 /user/logout 返回 404。两种 URL 容易混淆，应统一。

### 2.5 购物车操作使用 GET 请求
**严重度**: High
**端点**: /cart/addOne, /cart/subOne, /cart/deleteCart, /cart/addToCart
**描述**: 所有购物车增删改操作使用 GET 请求。GET 请求应仅用于读取数据，修改操作应使用 POST。
**风险**: 浏览器预加载、搜索引擎爬虫可能意外触发操作；URL 可被轻易分享导致 CSRF 攻击。

### 2.6 空凭证登录无错误提示
**严重度**: High
**描述**: 使用空用户名和空密码 POST 登录，返回 HTTP 200 且仍然显示登录表单，无任何错误提示。
**影响**: 用户不知道登录失败的原因。

## 三、中危 Bug (Medium)

### 3.1 搜索空关键词返回全部商品
**严重度**: Medium
**端点**: GET /goods/search?keyword=
**描述**: 关键词为空或仅含空格时，返回完整商品列表（17692B），应提示"请输入搜索关键词"。

### 3.2 登录/注册页缺少导航栏
**严重度**: Medium
**描述**: /user/login 和 /user/register 页面没有导航栏（navbar），登录失败的用户无法浏览商品，体验不连贯。

### 3.3 订单 URL 命名不一致
**严重度**: Medium
**描述**:
- 导航栏链接: /order/myOrder
- 可访问页面: /order/orderList
- 两者都返回 200 且内容不同（4838B vs 4149B），存在功能分歧。

### 3.4 支付金额可被客户端篡改
**严重度**: Medium
**描述**: 结算表单中总金额通过隐藏字段 <input type="hidden" name="total" value="1948.0"> 传递，用户可修改为任意金额。
**修复建议**: 后端根据购物车内容重新计算总金额，忽略客户端传值。

### 3.5 响应式断点仅 768px
**严重度**: Medium
**描述**: CSS 仅有 1 个媒体查询 @media(max-width:768px)，缺少平板（1024px）和大屏适配，移动端体验可能不佳。

## 四、低危 Bug (Low)

### 4.1 评论功能未实现
**严重度**: Low
**描述**: 商品详情页有评论区域（#reviewSection），但后端 /review/add 返回 400，/comment/add 返回 404，功能未完成。

### 4.2 注册表单无密码确认
**严重度**: Low
**描述**: 注册表单只有单个密码字段，缺少"确认密码"字段，用户可能输错密码不自知。

### 4.3 密码无最小长度限制
**严重度**: Low
**描述**: 注册时密码可为任意长度（包括1位），应设置最小长度（如6位）。

### 4.4 缺少登录频率限制
**严重度**: Low
**描述**: 登录接口无验证码或频率限制，存在暴力破解风险。

### 4.5 Cart 操作错误处理不一致
**严重度**: Low
**描述**:
- cartId=abc -> 400（有验证）
- cartId=-1 -> 200（无验证，接受负数）
- 验证逻辑不一致

### 4.6 Google Fonts 加载可能受阻
**严重度**: Low
**描述**: CSS 使用 @import url('https://fonts.googleapis.com/...') 加载 Playfair Display 和 DM Sans 字体。在国内网络环境下 Google Fonts 无法访问，导致字体回退到系统默认字体。
